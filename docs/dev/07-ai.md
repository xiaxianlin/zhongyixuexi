## 1. 概述

### 1.1 职责

AI 模块是本软件**唯一联网**的模块，负责对已解析入库的内容做"增强"而非"解析"：

- **文本生成（默认 DeepSeek）**：白话解读（AI-01）、智能问答 RAG（AI-02）、记忆卡批量生成（AI-06）、结构化标注（AI-04）。
- **可插拔厂商接口（DeepSeek 不支持，需另配）**：配图生成（AI-03）、TTS 朗读（AI-05）。
- **缓存与成本控制**：`ai_cache` 表缓存生成结果，按 `prompt_hash` 命中避免重复计费。
- **失败降级（AI-07）**：API Key 未配 / 调用失败 / 超时时，AI 入口明确提示并降级为本地纯文本，不阻断阅读与学习。
- **红线拦截**：System Prompt 显式禁止诊疗/剂量建议，问答端关键词兜底过滤。

### 1.2 边界

- **不做**：导入解析（IMP）、阅读排版（RD）、记忆卡调度（LRN）、全文索引构建（SRH）。本模块只消费它们提供的段落/检索结果，产出文本/卡片/音频/图片。
- **不做**：账号体系、云同步、服务端。所有生成内容本地缓存。
- **联网范围**：仅主进程 HTTP 客户端向用户配置的 AI 端点（默认 DeepSeek）发请求；渲染进程无网络权限。

### 1.3 与其它模块的关系

| 模块 | 关系 |
|---|---|
| **SET（设置）** | 依赖 SET-01：API Key 经 `safeStorage` 加密存 `settings` 表，本模块读取解密后使用；厂商/模型/端点配置也来自 SET。 |
| **IMP（导入解析）** | 依赖 `paragraphs` / `chapters` 表的稳定 ID 与正文，作为生成的输入与绑定锚点。 |
| **RD（阅读）** | 为 RD-03 解读栏、RD-04 资源栏（配图）、RD-05 词条浮窗提供内容。 |
| **SRH（检索）** | 复用 `fts_paragraphs`（FTS5）做 RAG 的 top-k 段落检索；可选 `sqlite-vec` 语义检索。 |
| **LRN（学习）** | AI-06 批量生成卡片草稿写入 `cards`（待用户确认）；标注结果供 LRN 抽卡。 |
| **NOTE（笔记）** | AI 问答答案带 `paragraph_id` 来源，可一键转引用写入笔记。 |

## 4. 数据模型

### 4.1 `ai_cache` 表（核心）

存储所有 AI 文本生成结果，按 `prompt_hash` 命中以避免重复计费。

```sql
-- electron/db/schema/07-ai-cache.sql
CREATE TABLE IF NOT EXISTS ai_cache (
  id            TEXT    PRIMARY KEY,            -- UUID v4
  scope         TEXT    NOT NULL,               -- 命中范围：'paragraph' | 'chapter' | 'global'
  scope_id      TEXT    NOT NULL,               -- paragraph_id / chapter_id / 'qa' 等
  kind          TEXT    NOT NULL,               -- 'modern' | 'explanation' | 'qa' | 'cards' | 'annotation'
  prompt_hash   TEXT    NOT NULL,               -- sha256(normalized_prompt)，命中键
  response      TEXT    NOT NULL,               -- 原始/结构化结果（JSON 字符串或纯文本）
  model         TEXT    NOT NULL,               -- 'deepseek-chat' 等
  prompt_tokens INTEGER NOT NULL DEFAULT 0,
  completion_tokens INTEGER NOT NULL DEFAULT 0,
  total_tokens  INTEGER NOT NULL DEFAULT 0,
  created_at    INTEGER NOT NULL,               -- unix ms
  invalidated   INTEGER NOT NULL DEFAULT 0,     -- 0 有效 / 1 用户手动失效（重生成时置 1 旧记录）
  meta          TEXT                             -- JSON：附加信息（如 qa 的 query、cards 的 count）
);

-- 命中查询：相同 scope_id + kind + prompt_hash 且未失效
CREATE INDEX IF NOT EXISTS idx_ai_cache_hit
  ON ai_cache(scope_id, kind, prompt_hash, invalidated);

CREATE INDEX IF NOT EXISTS idx_ai_cache_scope
  ON ai_cache(scope, scope_id, kind);
```

**命中策略**：

1. 生成前计算 `prompt_hash = sha256(system_prompt_hash + user_prompt_normalized + model + temperature)`。`normalized` 指去除纯空白差异、统一换行后的稳定文本。
2. 查询 `SELECT ... WHERE scope_id=? AND kind=? AND prompt_hash=? AND invalidated=0 ORDER BY created_at DESC LIMIT 1`。
3. 命中则直接返回 `response`，**不计费、不联网**。
4. 未命中才调用 DeepSeek，成功后写库。
5. **手动失效重生成**：用户点"重新生成"→ 将当前 scope_id+kind 下所有记录 `invalidated=1` → 重新走生成流程（不删除旧记录，保留历史便于审计/回退）。
6. 段落正文被用户编辑（IMP-03）后，因 `paragraph_id` 稳定但文本变化，`prompt_hash` 自然不同，会生成新版而旧版仍保留（避免误命中过期解读）。

### 4.2 关联（复用既有表）

- `paragraphs.content_modern` / `paragraphs.content_explanation`：AI-01 生成结果同时**回写段落字段**（便于阅读栏不查缓存表直接渲染），`ai_cache` 作为权威/历史源。回写与缓存写入在同一事务。
- `cards`（LRN 表）：AI-06 生成的卡片草稿带 `source='ai'`、`confirmed=0`，用户确认后置 `confirmed=1` 进入调度。
- `settings`（SET 表）：键 `ai.provider`、`ai.model`、`ai.endpoint`、`ai.api_key_enc`（safeStorage 密文）、`ai.temperature` 等。

## 5. IPC 接口

channel 前缀 `ai:*`。长任务（生成）用 `webContents.send('ai:progress', payload)` 推进度。

| channel | 入参 | 返回 | 长任务 | 说明 |
|---|---|---|---|---|
| `ai:status` | — | `{ configured: boolean, provider: string, model: string }` | 否 | 探测 Key 是否已配（不解密，仅判断存在） |
| `ai:generateModern` | `{ paragraphId: string }` | `ModernResultDTO` | 是 | AI-01：单段白话解读（命中缓存则秒回） |
| `ai:generateModernBatch` | `{ chapterId: string }` | `{ done: number, total: number }` | 是 | AI-01 批量：整章逐段生成，逐段推进度 |
| `ai:ask` | `{ query: string, bookId?: string, topK?: number }` | `QaAnswerDTO` | 是 | AI-02 RAG 问答，返回答案 + 来源段落列表 |
| `ai:generateCards` | `{ paragraphIds: string[] }` | `CardDraftDTO[]` | 是 | AI-06 生成卡片草稿（不入库，前端确认后调 LRN 写库） |
| `ai:annotate` | `{ paragraphId: string }` | `AnnotationDTO` | 是 | AI-04 抽取关键词/实体/属性（P2） |
| `ai:generateImage` | `{ prompt: string, paragraphId?: string }` | `{ assetId: string }` | 是 | AI-03 配图（P2，需另配图像厂商，未配则返回 `AI_PROVIDER_NOT_CONFIGURED`） |
| `ai:tts` | `{ text: string, voice?: string, speed?: number }` | `{ assetId: string }` | 是 | AI-05 TTS（P2，需另配） |
| `ai:invalidate` | `{ scopeId: string, kind: string }` | `{ invalidated: number }` | 否 | 手动失效缓存（触发重生成） |
| `ai:progress` | （主进程推送） | `{ jobId: string, phase: string, current: number, total: number }` | — | 长任务进度事件 |

**错误返回**：所有 handler 抛 `AiError`（`{ code, message, details? }`，见 §8），经 IPC 序列化，渲染进程 `try/catch` 捕获并按 `code` 降级。

**参数约束**：`paragraphId` 必须存在且属于当前库；`query` 长度 ≤ 2000 字符；`topK` 默认 5、上限 10。

## 6. 核心设计

### 6.1 DeepSeek 客户端封装

主进程用全局 `fetch`（Node 18+ 内置 undici）调用 DeepSeek `chat/completions`，统一请求/重试/超时/错误映射。

```ts
// electron/ai/client/types.ts
export interface ChatMessage {
  role: 'system' | 'user' | 'assistant';
  content: string;
}

export interface ChatRequest {
  model: string;            // 'deepseek-chat'
  messages: ChatMessage[];
  temperature?: number;     // 0.3-0.5（解读/标注），0.5-0.7（问答）
  max_tokens?: number;
  response_format?: { type: 'json_object' };  // JSON mode（解读/卡片/标注）
  stream?: false;           // 本模块默认非流式（简化缓存与进度）
}

export interface ChatResponse {
  id: string;
  choices: { message: ChatMessage; finish_reason: string }[];
  usage: { prompt_tokens: number; completion_tokens: number; total_tokens: number };
  // DeepSeek 兼容 OpenAI 格式
}
```

```ts
// electron/ai/client/deepseek.ts
import { AiError } from '../errors';

const DEFAULT_ENDPOINT = 'https://api.deepseek.com/chat/completions';
const DEFAULT_MODEL = 'deepseek-chat';
const TIMEOUT_MS = 10 * 60_000;
const MAX_RETRIES = 2;          // 共 1+2=3 次
const RETRYABLE_STATUS = new Set([408, 429, 500, 502, 503, 504]);

export interface DeepSeekConfig {
  apiKey: string;
  endpoint?: string;            // 用户可改
  model?: string;
}

export async function chat(
  req: ChatRequest,
  cfg: DeepSeekConfig,
  signal?: AbortSignal,
): Promise<ChatResponse> {
  const url = cfg.endpoint || DEFAULT_ENDPOINT;
  const timeoutCtl = new AbortController();
  const t = setTimeout(() => timeoutCtl.abort(), TIMEOUT_MS);
  // 组合外部 signal
  if (signal) signal.addEventListener('abort', () => timeoutCtl.abort());

  let lastErr: unknown;
  for (let attempt = 0; attempt <= MAX_RETRIES; attempt++) {
    try {
      const res = await fetch(url, {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          Authorization: `Bearer ${cfg.apiKey}`,
        },
        body: JSON.stringify(req),
        signal: timeoutCtl.signal,
      });

      // 401/403：Key 问题，不重试
      if (res.status === 401 || res.status === 403) {
        throw new AiError('AI_AUTH_ERROR',
          `API 鉴权失败（${res.status}），请检查 API Key 配置`, { status: res.status });
      }
      // 余额/配额
      if (res.status === 402 || res.status === 429) {
        const body = await res.text();
        throw new AiError('AI_QUOTA_EXCEEDED',
          res.status === 429 ? '请求频率超限，请稍后重试' : 'API 余额不足或配额用尽',
          { status: res.status, body });
      }
      // 可重试的服务端错误
      if (RETRYABLE_STATUS.has(res.status)) {
        lastErr = new AiError('AI_SERVER_ERROR',
          `AI 服务暂时不可用（${res.status}）`, { status: res.status });
        await sleep(backoffMs(attempt));   // 指数退避 500/1000/2000
        continue;
      }
      if (!res.ok) {
        const body = await res.text();
        throw new AiError('AI_REQUEST_FAILED',
          `AI 请求失败（${res.status}）`, { status: res.status, body });
      }
      return (await res.json()) as ChatResponse;
    } catch (e) {
      if (e instanceof AiError && !RETRYABLE_STATUS.has((e.details as any)?.status)) throw e;
      lastErr = e;
      if (attempt < MAX_RETRIES) await sleep(backoffMs(attempt));
    }
  } finally {
    clearTimeout(t);
  }
  // 超时
  if (lastErr instanceof DOMException && lastErr.name === 'AbortError') {
    throw new AiError('AI_TIMEOUT', `AI 请求超时（${TIMEOUT_MS}ms）`);
  }
  throw lastErr instanceof AiError ? lastErr
    : new AiError('AI_UNKNOWN', 'AI 调用失败', { cause: String(lastErr) });
}

const sleep = (ms: number) => new Promise(r => setTimeout(r, ms));
const backoffMs = (attempt: number) => 500 * Math.pow(2, attempt);  // 500/1000/2000
```

**API Key 读取（经 SET / safeStorage 解密）**：

```ts
// electron/services/ai.ts（片段）
import { safeStorage } from 'electron';

function loadConfig(): DeepSeekConfig {
  const row = db.prepare(
    "SELECT value FROM settings WHERE key IN ('ai.api_key_enc','ai.endpoint','ai.model')"
  ).all() as { key: string; value: string }[];
  const m = Object.fromEntries(row.map(r => [r.key, r.value]));
  const enc = m['ai.api_key_enc'];
  if (!enc) throw new AiError('AI_KEY_NOT_CONFIGURED', '未配置 API Key，请在设置中添加');
  if (!safeStorage.isEncryptionAvailable()) {
    throw new AiError('AI_STORAGE_UNAVAILABLE', '系统不支持安全存储，无法解密 API Key');
  }
  const apiKey = safeStorage.decryptString(Buffer.from(enc, 'base64'));
  return {
    apiKey,
    endpoint: m['ai.endpoint'] || undefined,
    model: m['ai.model'] || DEFAULT_MODEL,
  };
}
```

> **安全约束**：Key 密文以 base64 存库；解密仅在主进程内存中瞬时使用；**禁止**写入日志、禁止回传渲染进程、禁止出现在错误 `details` 中（仅状态码/通用消息）。

#### 6.2.2 AI-01 白话解读（modern.ts）—— 逐句结构化 JSON

约束：`temperature: 0.3`，`response_format: json_object`。

```
[System]
<红线片段>
你的任务：把给定的中医古籍原文逐句翻译为白话，并给出医理点拨。
仅基于原文含义翻译与解释，不得添加原文没有的诊断或用药主张。

[User]
原文：
"""
人参，味甘微寒。主补五脏，安精神，定魂魄，止惊悸，除邪气，明目，开心益智。
"""

请严格输出如下 JSON（不要输出 JSON 以外的任何文字）：
{
  "version": 1,
  "sentences": [
    {
      "original": "原文单句（按句号/逗号切分，保持顺序）",
      "modern": "对应白话译文，通俗准确",
      "commentary": "医理点拨：解释该句的中医理论背景、关键术语含义（1-2句，不得给出剂量/处方）"
    }
  ],
  "summary": "整段一句话概括（≤40字）"
}

规则：
- sentences 数量与原文句数对应，顺序一致；
- modern 为现代汉语白话，不照搬古文；
- commentary 仅解释医理与术语，禁止诊疗/剂量；
- 若原文为残篇/存疑，commentary 标注"原文存疑"。
```

**校验（schema.ts）**：解析 JSON → 校验 `sentences[].original` 与原文切分句数一致（容差 ±1）→ 非空字段检查 → 失败则重试一次（降温度 0.2）→ 仍失败抛 `AI_PARSE_ERROR`。

**输出 DTO**：

```ts
interface ModernResultDTO {
  paragraphId: string;
  fromCache: boolean;
  sentences: { original: string; modern: string; commentary: string }[];
  summary: string;
  model: string;
  tokens: number;
}
```

#### 6.2.3 AI-02 问答 RAG（qa.ts）

约束：`temperature: 0.5`，非 JSON mode（自然语言答案 + 末尾 JSON 引用块，正则解析）。

```
[System]
<红线片段>
你是一个基于"检索到的本书段落"回答问题的学习助手。
规则：
1. 仅依据下方【检索段落】作答，不得编造未给出的内容；
2. 每个事实陈述须标注来源序号，形如 [1]、[2]，对应检索段落编号；
3. 若检索段落不足以回答，回答"根据现有内容无法回答该问题"，不要臆测；
4. 严格遵守红线：不诊疗、不处方、不给剂量。

[User]
【检索段落】
[1] （《XX》第N章）人参，味甘微寒，主补五脏……（paragraph_id=p_abc1）
[2] （《XX》第M章）黄芪，味甘微温，补气固表……（paragraph_id=p_def2）
...（top-k 段落，每段标注序号、出处、paragraph_id）

【问题】
{query}

请输出：
1. 先用 2-4 句自然语言回答（带 [n] 引用）；
2. 末尾另起一行，输出 JSON：{"cites":[{"n":1,"paragraph_id":"p_abc1","snippet":"人参，味甘微寒..."}]}
```

**校验/降级**：解析末尾 JSON 失败时不阻断，答案正文仍展示，仅来源列表为空并提示"未能解析来源"。

#### 6.2.4 AI-06 记忆卡批量生成（cards.ts）

约束：`temperature: 0.4`，`response_format: json_object`。

```
[System]
<红线片段>
任务：从给定原文抽取要点，生成"问答型记忆卡"草稿，供学习者背诵复习。
卡片应为原文知识点的忠实提炼，不得添加剂量/处方类信息。

[User]
原文：
"""
{paragraph_text}
"""

输出 JSON：
{
  "cards": [
    {
      "front": "问题/正面提示（如术语、原文上句）",
      "back": "答案/释义（对应原文要点，白话）",
      "tag": "术语|原文|功效|性味|其他"
    }
  ]
}
规则：每段生成 2-5 张卡；front 简洁；back 不超过 60 字；禁止诊疗/剂量。
```

草稿不直接入库，前端 `CardsGenDialog` 让用户勾选确认，再调 LRN 的 `cards:create` 写入。

#### 6.2.5 AI-04 结构化标注（annotation.ts，P2）

约束：`temperature: 0.3`，JSON mode。抽取关键词/实体；中医典籍标注性味/归经/功效属性。

```json
{
  "keywords": [{"text": "人参", "type": "药味"}],
  "attributes": [{"entity": "人参", "nature": "微寒", "flavor": "甘", "meridians": ["脾","肺"], "effects": ["补五脏","安精神"]}]
}
```

> 属性仅作学习参考，非权威；前端标注"AI 辅助生成"。

### 6.3 RAG 智能问答流程（AI-02）

```
用户在问答框输入 query
        │
        ▼
[1] 关键词预检（guard.ts）：是否触发红线词（诊疗/剂量/处方）？
        │ 是 → 直接返回固定拒绝文案（不联网，不进缓存）
        │ 否 ↓
[2] 本地检索 top-k 段落：
     - 默认 FTS5：SELECT ... FROM fts_paragraphs WHERE fts_paragraphs MATCH ? 
       ORDER BY bm25(fts_paragraphs) LIMIT {topK}（query 做 AND/OR 分词）
     - 可选 sqlite-vec：query 向量化（需 embedding 厂商，Phase 2）→ 余弦 top-k
     - 可限定 bookId 范围
        ▼
[3] 拼装 Prompt（§6.2.3）：检索段落编号 + paragraph_id 注入
        ▼
[4] 查 ai_cache（kind='qa', prompt_hash 命中？）
     命中 → 直接返回（不联网）
        ▼
[5] DeepSeek chat（temperature 0.5）→ 解析答案 + cites JSON
        ▼
[6] guard.ts 后置过滤：答案正文再次扫描红线关键词，命中则净化/拦截
        ▼
[7] 写 ai_cache，返回 QaAnswerDTO（答案 + 来源段落列表，前端可点 [n] 跳转到 paragraph_id）
```

**来源跳转**：`cites[].paragraph_id` 传给阅读模块，定位到对应段落并高亮 `snippet` 片段（复用 RD 段落定位能力）。

### 6.4 长任务进度推进

整章批量生成（`ai:generateModernBatch`）等长任务：

```ts
// electron/services/ai.ts（片段）
async function generateModernBatch(chapterId: string, sender: WebContents) {
  const paragraphs = listParagraphs(chapterId);
  const jobId = uuid();
  let done = 0;
  for (const p of paragraphs) {
    sender.send('ai:progress', { jobId, phase: 'modern', current: done, total: paragraphs.length });
    await generateModern(p.id);   // 单段：查缓存→命中秒过→否则调 API
    done++;
  }
  sender.send('ai:progress', { jobId, phase: 'done', current: done, total: done });
  return { done, total: paragraphs.length };
}
```

渲染进程 `stores/ai.ts` 监听 `ai:progress` 更新进度条；用户可关闭进度条，生成继续在主进程后台完成（结果写库，下次打开可见）。

### 7.2 store 结构（stores/ai.ts，Zustand）

```ts
interface AiStore {
  status: { configured: boolean; provider: string; model: string } | null;
  degraded: boolean;                 // 是否处于降级态
  degradedReason?: string;           // 降级原因（i18n key）
  generating: Record<string, AiProgress>;  // jobId → 进度
  // actions
  refreshStatus: () => Promise<void>;
  onProgress: (p: AiProgress) => void;
  enterDegraded: (reason: string) => void;
  exitDegraded: () => void;
}
```

### 7.3 关键交互与状态流转（AI-07 降级）

```
渲染进程调用 window.api.ai.*
        │
   try/catch AiError
        │
   按 code 分支：
   - AI_KEY_NOT_CONFIGURED → degradedReason='key_missing'，显示 DegradedNotice + 解读栏显示"前往设置配置 API Key"按钮
   - AI_AUTH_ERROR / AI_QUOTA_EXCEEDED → degradedReason='auth_or_quota'，提示检查 Key/余额
   - AI_TIMEOUT / AI_SERVER_ERROR → degradedReason='network'，提示"AI 服务暂时不可用，已切换本地模式"
   - 任何 AI 错误 → degraded=true，但阅读/记忆/检索/笔记照常可用
        │
   降级态下：
   - AI 入口（解读栏/问答框）禁用或显示提示，不抛错打断用户；
   - 用户在设置配好 Key 或网络恢复后，AiStatusBadge 自动恢复（定时探活或设置页保存后触发 ai:status）。
```

**原则**：AI 故障绝不阻断核心阅读/学习路径；解读栏为空时阅读栏原文照常显示。

## 8. 错误处理与边界

### 8.1 AiError 错误码

| code | 含义 | 降级动作 |
|---|---|---|
| `AI_KEY_NOT_CONFIGURED` | 未配置 API Key | 提示去设置；降级本地纯文本 |
| `AI_STORAGE_UNAVAILABLE` | safeStorage 不可用（如 Linux 无 keyring） | 提示系统不支持加密存储 |
| `AI_AUTH_ERROR` | 401/403 鉴权失败 | 提示检查 Key 正确性/厂商 |
| `AI_QUOTA_EXCEEDED` | 402 余额/429 限流 | 提示余额或稍后重试 |
| `AI_TIMEOUT` | 请求超时（10 分钟） | 提示网络/服务慢，降级 |
| `AI_SERVER_ERROR` | 5xx 等可重试错误（重试耗尽） | 提示服务不可用，降级 |
| `AI_REQUEST_FAILED` | 其他 HTTP 错误 | 提示失败，降级 |
| `AI_PARSE_ERROR` | 模型输出 JSON 解析/校验失败 | 重试一次（降温度）；仍失败提示"生成失败，请重试" |
| `AI_GUARD_BLOCKED` | 触发红线拦截（用户问诊疗/剂量） | 返回固定拒绝文案，不报错 |
| `AI_PROVIDER_NOT_CONFIGURED` | 图像/TTS 厂商未配（P2） | 提示需在设置配置对应厂商 |
| `AI_UNKNOWN` | 未分类异常 | 通用失败提示，降级 |

> **日志安全**：`AiError.details` 仅含 HTTP 状态码、通用原因，**不含 API Key、不含完整响应体中的敏感头**。日志统一 `electron-log`，级别：失败为 `warn`，未预期为 `error`。

### 8.2 失败降级流程（AI-07，P0）

1. 渲染进程所有 AI 调用包裹 `try/catch`。
2. 捕获 `AiError` → `stores/ai.enterDegraded(code)` → 顶部 `DegradedNotice` 横幅 + 对应面板显示降级提示。
3. **核心路径不受影响**：原文栏、记忆卡翻卡、全文检索、笔记编辑全部本地可用。
4. 设置页保存 Key 后 → 调 `ai:status` 刷新 → 若 `configured=true` → `exitDegraded()`。

### 8.3 边界条件

- **空段落/超长段落**：空段落不生成（前端拦截）；超长（> 模型上下文一半）按句切分多次生成后合并，每段独立缓存。
- **并发**：同一 `scope_id+kind` 并发生成用内存级 `inflight` Map 去重（同请求只发一次，余者 await 同一 Promise）。
- **离线**：`navigator.onLine` 与主进程探活结合；离线时 AI 入口直接降级，不发起 doomed 请求。
- **Key 含特殊字符**：透传到 `Authorization` 头，不做拼接 SQL（Key 不进 SQL，仅内存使用）。

## 9. 红线与合规设计

### 9.1 三层拦截

1. **System Prompt 硬约束**（§6.2.1）：所有模板强制拼接红线片段，明确禁止诊疗/剂量/处方。

- 应用开屏与阅读页底部常驻免责声明（SET-05）：本软件为阅读学习工具，不构成医疗建议。

## 13. 开放问题

1. **向量检索首期是否引入**：FTS5 关键词检索已能满足多数中医典籍 RAG；`sqlite-vec` 语义检索（需额外 embedding 厂商/本地模型）是否在 Phase 2 引入？PRD §13-3 建议 FTS5 起步。**当前决策**：首期仅 FTS5，预留 `rag.ts` 向量分支接口。
2. **流式输出**：当前非流式（简化缓存与进度）。问答框是否改为 SSE 流式以提升体感？需评估 DeepSeek 流式 + 缓存一致性。**当前决策**：P0 非流式，后续问答场景可单独引入流式（缓存以完整结果落库）。
3. **配图/TTS 默认厂商**：DeepSeek 不支持，AI-03/05 为 P2。是否预设某家（如即梦图像、字节 TTS）作为"推荐配置"降低用户门槛？PRD §13-4 待定。**当前决策**：仅留可插拔接口，不预设默认厂商。
4. **token 预估与限额**：是否在调用前向用户展示预估 token 消耗并设软上限（避免误触大额调用）？PRD §7.2 提及预估提示。**当前决策**：批量生成时在 UI 提示"将生成 N 段，预计消耗 ~M tokens"，单次生成不强制确认。
5. **模型升级/切换**：`ai.model` 变更后，旧 `prompt_hash` 因含 model 维度不会误命中旧模型结果——符合预期（换模型应重新生成），但需考虑是否提供"一键全量重生成"。
